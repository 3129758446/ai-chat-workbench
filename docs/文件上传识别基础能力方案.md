# 文件上传识别基础能力方案

## 任务目标

在现有图片上传能力基础上，新增轻量级文本类文件上传识别能力，使用户可以上传文件，并在当前会话中基于文件内容继续提问。

本任务只做前端侧基础文件读取、文本抽取、会话绑定和消息上下文拼接，不做完整 RAG，不接向量数据库，不新增后端文件存储服务。

## 优化范围

本阶段允许优化的范围如下：

1. 复用现有上传入口，在图片上传能力旁边扩展文本类文件选择能力。
2. 在前端读取受支持文件的文本内容，并生成可被模型理解的上下文片段。
3. 将已解析文件绑定到当前会话，保证会话切换后文件上下文互不串扰。
4. 在发送消息时，把当前会话内可用的文件内容与用户问题一起发送给模型。
5. 增加上传、解析、成功、失败、移除等基础状态展示。
6. 增加文件大小、数量、类型和解析失败的前端校验。
7. 保持现有图片上传、文本聊天、流式回复、多会话、主题切换、清空会话等功能不受影响。

## 不允许改动的范围

本任务明确不做以下内容：

1. 不接入向量数据库、全文索引、embedding 或完整 RAG 流程。
2. 不新增服务端文件上传、文件持久化、对象存储或后端解析服务。
3. 不改变现有聊天接口协议的核心结构，仅在前端构造用户消息内容时追加文件上下文。
4. 不移除或重写现有图片上传能力。
5. 不改变现有会话路由结构 `/chat/:conversationId`。
6. 不改变现有 API Key 读取、模型请求、流式输出和错误处理主链路。
7. 不引入大型解析库或明显增加首屏体积的依赖。
8. 不支持二进制文档深度解析，例如 Word 复杂排版、扫描版 PDF OCR、Excel 多表格语义识别。
9. 不保证超长文件完整进入模型上下文，必须做长度限制和截断提示。

## 支持文件类型

首阶段只支持浏览器端可以稳定、轻量解析的文本类文件：

| 类型 | 扩展名 | MIME 参考 | 解析方式 |
| --- | --- | --- | --- |
| 纯文本 | `.txt` | `text/plain` | `File.text()` |
| Markdown | `.md`, `.markdown` | `text/markdown`, `text/plain` | `File.text()` |
| JSON | `.json` | `application/json` | `File.text()` 后按文本使用，可做格式化校验 |
| CSV | `.csv` | `text/csv`, `application/vnd.ms-excel` | `File.text()` |
| 日志 | `.log` | `text/plain` 或空 MIME | `File.text()` |
| 代码文本 | `.js`, `.ts`, `.tsx`, `.jsx`, `.css`, `.html`, `.xml`, `.yaml`, `.yml` | 常见 text 或空 MIME | `File.text()` |

暂不支持：

1. `.pdf`
2. `.doc`, `.docx`
3. `.xls`, `.xlsx`
4. 图片 OCR
5. 压缩包
6. 音频、视频
7. 加密文件或无法按文本读取的二进制文件

后续如需支持 PDF 或 Office，应作为独立阶段评估解析库体积、准确性、性能和安全边界。

## 功能要求

### 1. 上传入口

1. 保留现有图片上传入口和交互。
2. 可以采用同一个上传按钮，但文件选择 `accept` 需要同时覆盖 `image/*` 与受支持文本文件。
3. 也可以拆成图片上传与文件上传两个按钮，但必须保持输入区简洁，不影响原发送按钮、停止按钮和清空会话按钮。
4. 选择文件后，前端按类型区分图片和文本类文件：
   - 图片沿用现有 `uploadingImages` 流程。
   - 文本文件进入新增的待解析/已解析文件流程。

### 2. 文件解析

1. 使用浏览器 `File.text()` 或 `FileReader` 在前端读取内容。
2. 解析过程应是异步的，不能阻塞发送区交互。
3. 文件内容需保留文件名、类型、大小、解析时间和文本内容。
4. 解析成功后展示文件卡片，至少显示文件名、大小、状态和移除按钮。
5. 解析失败后展示明确错误，不应让失败文件进入可发送上下文。
6. 空文件应提示“文件内容为空”，不进入可发送上下文。

### 3. 消息发送

1. 允许以下发送组合：
   - 纯文本问题
   - 图片 + 文本问题
   - 文件 + 文本问题
   - 图片 + 文件 + 文本问题
   - 仅文件
2. 仅文件发送时，需要自动生成兜底问题，例如“请总结我上传的文件内容。”
3. 文件上下文应拼接为明确结构，例如：

```text
以下是用户上传的文件内容，请基于这些内容回答问题。

[文件 1]
文件名：example.md
类型：text/markdown
内容：
...

用户问题：
...
```

4. 多文件按用户上传顺序拼接。
5. 如果同时存在图片和文件，应保持现有多模态图片结构，同时把文件内容作为文本片段放在图片前或用户问题前，避免破坏视觉模型输入。
6. 发送成功进入请求流程后，应清空当前会话的待发送文件草稿；已发送文件内容是否保留在历史中，由消息内容决定。

### 4. 展示要求

1. 待发送文件应出现在现有上传预览区或其相邻区域。
2. 文件预览不展示完整正文，避免输入区被撑开。
3. 文件卡片需要有明确状态：
   - 解析中
   - 已就绪
   - 解析失败
   - 已截断
4. 文件卡片需要支持移除。
5. 错误提示应就近展示，不使用只在控制台输出的方式。
6. 移动端下文件卡片不能挤压输入框和主要操作按钮。

## 边界要求

### 1. 文件大小限制

建议首阶段限制：

1. 单文件最大 `1 MB`。
2. 单会话待发送文件最多 `3` 个。
3. 单次发送进入模型的文件总文本上限建议为 `20,000` 字符。

超过限制时：

1. 文件过大：拒绝解析并提示。
2. 文件数量过多：拒绝新增并提示。
3. 内容过长：允许解析，但发送前截断，并在消息中明确标注“内容已截断”。

### 2. 文本长度与上下文污染

1. 文件内容不能无限追加到每次请求。
2. 已发送文件内容会进入该轮 `chatHistory`，后续是否继续作为上下文依赖现有历史机制。
3. 不额外实现跨轮文件检索、召回或按需片段选择。
4. 清空会话时必须同时清空该会话内未发送的文件草稿和解析状态。

### 3. 安全与兼容

1. 不执行上传文件中的任何代码。
2. 不把文件内容写入 DOM 的 `innerHTML`。
3. 不信任 MIME，需结合扩展名和 MIME 做双重判断。
4. 对无法识别编码导致的乱码，给出解析失败或内容异常提示。
5. 文件内容只存在浏览器内存和当前前端持久化策略允许的会话消息中，不上传到额外服务。

## 会话管理绑定

新增文件能力必须跟随现有多会话模型设计：

1. 每个 `Conversation` 增加独立的待上传文件列表，例如 `uploadingFiles`。
2. `uploadingFiles` 只属于当前会话，切换会话时展示对应会话自己的文件草稿。
3. 首页上传文件时，应与现有首页图片草稿策略保持一致：
   - 如果首页只负责创建会话和跳转，则文件草稿应挂载到新建会话。
   - 跳转到聊天页后，文件草稿仍可继续发送。
4. 删除会话时，需要移除该会话的文件草稿和解析状态。
5. 清空会话时，需要清空该会话的文件草稿。
6. 持久化策略建议：
   - 未发送文件草稿不持久化，避免刷新后保留大文本和 File 对象。
   - 已发送的文件内容作为该轮用户消息的一部分进入 `chatHistory`，是否被持久化沿用现有消息历史持久化策略。
   - `File` 原始对象不持久化。

建议新增类型：

```ts
type UploadingFileStatus = "parsing" | "ready" | "error";

interface UploadingTextFile {
  id: string;
  file: File;
  name: string;
  size: number;
  type: string;
  extension: string;
  status: UploadingFileStatus;
  text: string;
  error?: string;
  truncated?: boolean;
  createdAt: number;
}
```

## 异步状态和错误处理

### 1. 解析状态

文件从选择到可发送至少经过以下状态：

1. `parsing`：正在读取文件。
2. `ready`：解析成功，可进入发送上下文。
3. `error`：解析失败，不参与发送。

发送按钮可用规则建议：

1. 有文本、图片或 `ready` 文件时可发送。
2. 只有 `parsing` 文件时不可发送，或发送时忽略解析中文件并提示用户等待。
3. 有 `error` 文件时不阻塞其他有效内容发送，但错误文件不进入上下文。

### 2. 错误类型

必须覆盖以下错误：

1. 文件类型不支持。
2. 文件过大。
3. 文件数量超过限制。
4. 文件读取失败。
5. 文件内容为空。
6. 文件内容过长并被截断。
7. 浏览器不支持当前读取能力。

### 3. 错误展示

1. 单文件错误展示在对应文件卡片上。
2. 批量选择时的整体错误可以展示在上传预览区顶部或输入区附近。
3. 错误文案必须能让用户知道下一步怎么做，例如“仅支持 txt、md、json、csv、log 和常见代码文本文件”。

## 不允许影响现有功能

实现时必须重点保护以下现有能力：

1. 纯文本发送不变。
2. 图片上传、预览、删除和发送不变。
3. 仅图片发送的兜底文本不变。
4. 视觉模型选择逻辑不变。
5. 流式输出、停止生成和错误回填不变。
6. 多会话创建、切换、删除、重命名不变。
7. 主题切换不变。
8. 本地 API Key 存取不变。
9. 页面刷新后的现有会话恢复行为不变。

## 分阶段实施计划

### 第一阶段：文档和边界确认

1. 明确支持类型、大小、数量、上下文长度限制。
2. 明确哪些状态进入会话，哪些状态不持久化。
3. 明确不做 RAG、不做服务端解析、不做 PDF/Office。

产出：本方案文档。

### 第二阶段：类型和状态模型

1. 在 `src/types/chat.ts` 增加文本文件上传类型。
2. 在 `Conversation` 中增加 `uploadingFiles`。
3. 在 `chatStore` 中增加：
   - 添加文件
   - 更新文件状态
   - 移除文件
   - 清空会话文件
4. 调整持久化白名单，避免持久化原始 `File` 对象和未发送文件草稿。

验收：会话切换时不同会话的文件草稿互不影响。

### 第三阶段：前端文件解析工具

1. 新增文件类型识别工具。
2. 新增文件大小、数量、扩展名校验。
3. 新增 `parseTextFile` 工具，统一返回成功或错误结果。
4. 增加截断工具，保证进入模型的内容不超过限制。

验收：受支持文件可解析，不支持文件能明确报错。

### 第四阶段：上传入口和预览 UI

1. 调整上传 `accept`。
2. 在 `App.tsx` 的文件选择逻辑中区分图片和文本文件。
3. 在 `Composer` 上传预览区展示文件卡片。
4. 支持移除文件和展示解析状态。

验收：图片和文本文件都能被选择，原图片预览不回归。

### 第五阶段：发送链路集成

1. 扩展发送校验，支持 `ready` 文件作为可发送内容。
2. 在 `messageContent.ts` 或新增工具中构造文件上下文文本。
3. 发送时把文件上下文、用户输入和图片内容组合成模型请求内容。
4. 发送后清理待发送文件草稿。
5. 失败时不误删仍未发送的文件，除非已经进入发送快照。

验收：用户上传文件后可以提问，模型能基于文件内容回答。

### 第六阶段：回归验证

1. 验证纯文本发送。
2. 验证仅图片发送。
3. 验证图片 + 文本发送。
4. 验证文件 + 文本发送。
5. 验证图片 + 文件 + 文本发送。
6. 验证多会话切换、删除、清空。
7. 验证刷新后现有会话恢复不受影响。

## 验收标准

功能验收：

1. 用户可以上传 `.txt`、`.md`、`.json`、`.csv`、`.log` 和常见代码文本文件。
2. 上传后能看到文件卡片、大小、解析状态和移除入口。
3. 解析成功后，用户可以基于文件内容提问。
4. 仅上传文件不输入问题时，也能发送并触发总结类默认问题。
5. 多文件按上传顺序进入上下文。
6. 超长内容会被截断，并在发送上下文中标注。

边界验收：

1. 不支持类型会被拒绝并提示。
2. 超过大小限制会被拒绝并提示。
3. 空文件会被拒绝并提示。
4. 解析失败不会进入模型上下文。
5. 解析中文件不会导致页面卡死。

会话验收：

1. A 会话上传的文件不会出现在 B 会话。
2. 删除 A 会话后，其未发送文件草稿被清理。
3. 清空当前会话后，消息、历史和未发送文件草稿都被清理。
4. 首页上传后跳转聊天页时，文件草稿仍绑定到新建会话。

回归验收：

1. 原图片上传识别能力正常。
2. 原仅文本聊天正常。
3. 原流式回复和停止生成正常。
4. 原会话切换、重命名、删除正常。
5. 原主题切换正常。

## 建议测试清单

1. 上传一个 `txt` 文件并提问“总结这个文件”。
2. 上传一个 `md` 文件并提问“提取标题和待办事项”。
3. 上传一个 `json` 文件并提问“解释字段含义”。
4. 上传一个 `csv` 文件并提问“概括表格内容”。
5. 同时上传图片和 `txt`，确认图片能力仍可用。
6. 上传不支持的 `pdf`，确认被拒绝且提示明确。
7. 上传超过限制的大文件，确认被拒绝或截断策略符合预期。
8. 在 A 会话上传文件，切换到 B 会话，确认 B 会话没有该文件。
9. 删除带文件草稿的会话，确认无残留错误。
10. 刷新页面，确认现有会话恢复行为未被破坏。
