## 1. 项目一句话介绍

这是一个基于 React + TypeScript 构建的 AI 多会话聊天工作台，支持：

- 首页引导态与快捷提问
- 多会话创建、切换、重命名、删除
- 流式回复与停止生成
- Qwen/灵犀 与 DeepSeek 双模型接入
- 图片与文本文件上传
- Markdown 渲染、代码高亮、复制按钮
- 本地持久化恢复
- 测试与基础工程化流程

它不是单纯的“输入一个问题，拿一个回答”的页面，而是一个具备工作台属性的聊天前端：会话有独立上下文、独立草稿、独立上传资源、独立流式状态，并且能通过路由稳定访问。

## 2. 技术栈与职责划分

### 2.1 核心技术栈

- React 19：页面与组件化开发
- TypeScript：领域模型、状态边界与函数签名约束
- Zustand：多会话状态管理
- React Router 7：首页与聊天页路由编排
- Vite：开发服务器、构建与代理
- marked + DOMPurify + highlight.js：Markdown 渲染、安全净化、代码高亮
- Vitest + Testing Library + Playwright：单元测试、交互测试、E2E 测试
- ESLint + Husky + lint-staged：静态检查与提交前校验

### 2.2 为什么这样选

#### React

- 适合做组件分层和状态驱动渲染。
- 项目拆成欢迎区、会话侧栏、消息区、输入区后，边界会很清晰。

#### TypeScript

- 聊天域模型复杂，尤其是多会话、多模态输入和流式状态并存。
- 先定义类型再写实现，能显著降低状态流转错误。

#### Zustand

- 比 Redux 更轻量，适合中型项目。
- 状态和动作放在一起，适合把会话、消息、上传、运行态拆成多个模块。

#### React Router

- 当前会话和 URL 绑定后，刷新恢复、深链接访问、会话切换都会更自然。

#### Vite

- 开发体验好，适合本地调试。
- 代理能力强，适合前端通过同源路径访问大模型接口。

#### marked + DOMPurify + highlight.js

- 当前项目更适合“Markdown -> HTML 字符串 -> DOM 增强”的渲染管线。
- 流式聊天场景下，先快速转 HTML，再后置高亮和复制按钮增强，比较适合现有架构。

从职责上，这个项目可以拆成六层：

1. 入口与路由层
2. 页面编排层
3. 组件层
4. Hook 流程层
5. 状态管理层

- `src/main.tsx`
- `src/route/index.tsx`

职责：

- 挂载 React 根组件

设计特点：

- `App` 不直接处理复杂请求细节。
- 发送、文件上传、副作用、会话管理都下沉到 Hook。

## 4. 核心链路总览

### 4.1 链路一：首页提问 -> 跳转聊天页 -> 自动发送

流程如下：

1. 用户在首页输入问题。
2. 点击发送按钮后，`useSendMessage` 发现当前是 `home` 模式。
3. 创建一个新会话 ID。
4. 通过 `navigate("/chat/:conversationId", { state })` 跳转到聊天页。
5. `App.tsx` 中的 `useEffect` 读取 `location.state`。
6. 如果 `shouldAutoSend` 为真，则把草稿放入当前会话并自动触发发送。
7. 发送后用 `navigate(..., { replace: true, state: null })` 清除路由临时状态。

这个链路里最容易被问到的问题是：

- 为什么首页不直接发请求？
- 为什么要把自动发送放在聊天页处理？
- 为什么需要 token 去重？

答案是：

- 首页没有稳定的会话上下文，真实请求应该统一发生在聊天页。
- 聊天页天然绑定 `conversationId`，更适合承接历史、上传草稿和流式状态。
- 如果不做 token 去重，刷新页面时 `location.state` 可能被重复消费，导致消息重复发送。

### 4.2 链路二：聊天页发送消息 -> 流式渲染 -> 写回历史

流程如下：

1. 读取当前会话和输入内容。
2. 检查是否在流式中，避免重复发送。
3. 检查文本、图片、文件是否为空。
4. 选择 provider 和模型。
5. 构造用户消息内容。
6. 先插入 UI 用户消息。
7. 再插入模型历史消息。
8. 清空输入框和上传草稿。
9. 插入 assistant 占位消息。
10. 发起 SSE 请求。
11. 增量更新 assistant 消息文本。
12. 请求完成后，将 assistant 最终文本写入历史。

### 4.3 链路三：上传文件 -> 解析 -> 绑定到会话 -> 发送

流程如下：

1. 用户选择文件。
2. 先把文件分成图片和文本文件。
3. 如果当前不在聊天页，先创建会话并跳到聊天页。
4. 图片生成 `Object URL` 用于本地预览。
5. 文本文件做类型、大小、数量校验。
6. 文本文件先以 `parsing` 状态写入 Store。
7. 异步读取文本内容。
8. 解析成功则更新为 `ready`，失败则更新为 `error`。
9. 发送时把图片转成 `Data URL`，把文本文件拼成结构化 Prompt。

## 5. 路由与页面模式设计

### 5.1 路由结构

当前项目有两个主要路由：

```text
/
/chat/:conversationId
```

设计意图：

- `/`：首页，引导态，不承接真实聊天历史
- `/chat/:conversationId`：聊天页，承接真实会话上下文

这样做的好处：

1. 当前会话有稳定地址
2. 刷新页面可以恢复到指定会话
3. 首页和聊天页可以共用同一套发送逻辑，但职责不同

### 5.2 为什么用 `mode`

`App` 通过 `mode = "home" | "chat"` 区分页面行为。

这使得：

- 首页输入框可以有自己的临时状态 `homeInput`
- 聊天页输入框使用当前会话的 `draftInput`
- 同一个页面框架可以复用，而不是拆两个完全独立页面

### 5.3 路由状态透传

从首页跳聊天页时，项目没有直接把问题写进全局状态，而是通过 `location.state` 透传：

- `draftPrompt`
- `shouldAutoSend`

这是一个典型的 SPA 跨页面临时状态传递方案。

### 5.4 为什么需要 token 去重

聊天页中的 `useEffect` 会消费 `location.state`：

- 设置草稿
- 自动发送

但如果页面刷新、重渲染或某些场景下 `location.state` 仍然存在，`useEffect` 可能再次执行。

所以项目引入了 `routePromptTokenRef`：

- 先根据 `location.key + conversationId + draftPrompt + shouldAutoSend` 生成 token
- 如果 token 已处理过，就直接跳过

这就是典型的幂等控制。

## 6. 状态层设计

这一章是整个项目最重要的部分之一。

### 6.1 顶层状态长什么样

核心状态包括：

```ts
theme;
modelProvider;
currentConversationId;
orderedConversationIds;
conversations;
abortControllers;
```

其中最关键的是：

```ts
conversations: Record<string, Conversation>
orderedConversationIds: string[]
```

### 6.2 为什么用“字典 + 顺序索引”

不直接用数组管理全部会话，原因有三个：

1. 按会话 ID 读取和更新时，字典是 O(1)
2. 排序和数据存储解耦，更容易维护
3. 后续扩展置顶、归档、搜索更方便

### 6.3 Conversation 结构为什么这样设计

一个会话里不仅有消息，还包括：

- `title`
- `messages`
- `chatHistory`
- `draftInput`
- `uploadingImages`
- `uploadingFiles`
- `isStreaming`
- `lastMessagePreview`

这说明项目不是把“聊天”看成一个简单数组，而是把每个会话看成一个完整的工作单元。

### 6.4 UI 消息与 API 历史为什么分轨

项目中有两套消息：

#### `messages`

给 UI 展示用，特点：

- 有 `id`
- 适合占位消息
- 支持流式文本更新
- 支持图片展示

#### `chatHistory`

给模型请求用，特点：

- 更接近接口协议
- 只保留必要上下文
- 不掺杂 UI 层临时状态

这样设计的价值是：

- UI 可以自由做打字机效果、错误提示、占位消息
- 模型上下文保持纯净，不被展示层污染

### 6.5 持久化为什么单独做裁剪和恢复

项目不是把整个 Store 直接存到 `localStorage`，而是通过：

- `partializeChatState()`
- `mergePersistedChatState()`

来显式控制：

- 哪些字段可以存
- 哪些字段不能存
- 恢复时如何补齐默认值

不能直接存的包括：

- `File`
- `AbortController`
- `Object URL`
- 运行中的 `isStreaming`

这体现的是“持久化状态”和“运行态状态”分层建模。

## 7. Hook 层设计

### 7.1 `useConversationManager`

职责：

- 处理会话创建、选择、删除
- 统一路由跳转逻辑

为什么需要它：

- 删除当前会话后跳去哪里，这是一套页面级逻辑，不适合散落在组件里

### 7.2 `useSendMessage`

职责：

- 封装完整发送事务

它处理的事情包括：

- 首页模式跳转
- 聊天页发送
- 内容校验
- provider 选择
- API Key 读取
- 消息插入
- 流式占位
- 错误分类
- 中断回滚

为什么值得单独抽：

- 发送链路是整个项目最复杂的业务流程
- 如果写在 `App.tsx`，会让主页面文件变得难以维护

### 7.3 `useFileHandlers`

职责：

- 统一处理文件选择后的逻辑
- 文件分类
- 目标会话确定
- 文本文件异步解析
- 首页上传后的跳转

设计亮点：

- 文件总是先归属某个会话，再进入上传草稿状态
- 不会出现文件脱离会话上下文的情况

### 7.4 `useAppEffects`

职责：

- 统一页面副作用

包括：

- 主题同步
- 输入框高度自适应
- 自动滚动到底部
- 页面卸载时清理对象 URL

为什么单独抽：

- 这些 effect 是“必须发生但不属于业务主链路”的副作用
- 从 `App.tsx` 拆出后可读性更高

## 8. 服务层与流式解析

### 8.1 `api.ts` 负责什么

服务层负责：

- provider 区分
- 模型选择
- 端点选择
- 发起请求
- 解析 SSE
- 传输层错误回退

它不负责：

- UI 占位消息
- 输入框清空
- 错误提示文案展示

这就是服务层和页面层的职责边界。

### 8.2 为什么支持多 provider

当前支持：

- 灵犀 / Qwen
- DeepSeek

设计上先把 provider 统一抽象为：

```ts
type ChatProvider = "lingxi" | "deepseek";
```

这样做的好处：

- 上层只关心 provider
- 下层再根据 provider 决定模型和端点

### 8.3 模型是怎么自动切换的

核心逻辑是：

- provider 是 `deepseek`：固定使用 DeepSeek 模型
- provider 是 `lingxi`：
  - 消息里有图片时，走视觉模型
  - 没图片时，走普通文本模型

这是典型的“能力感知型模型选择”。

### 8.4 为什么只在传输层错误时回退

当前项目对端点回退比较克制：

- `Failed to fetch`
- `NetworkError`
- 代理异常

才会回退到下一个端点。

而像：

- 401/403
- 429
- 400

这类业务错误不会回退。

原因：

- 业务错误代表请求已经到达服务端，问题是真实的业务问题
- 如果继续回退，容易掩盖真实原因

### 8.5 SSE 是怎么解析的

核心过程：

1. 用 `fetch` 发请求
2. 调 `response.body.getReader()`
3. 用 `TextDecoder` 解码
4. 分行处理 `data:`
5. 遇到 `[DONE]` 结束
6. 解析 JSON，提取 `delta.content`
7. 累加成 `fullText`
8. 回调上层 `onDelta(fullText)`

注意这里回调给上层的不是“增量片段”，而是“当前完整文本”。

这样上层就不用重复做拼接了。

## 10. 文件上传与多模态消息构造

### 10.1 为什么图片和文本文件分两套逻辑

因为两者本质不同：

- 图片：直接用于视觉模型识别
- 文本文件：本质上还是文本，要注入 Prompt

### 10.2 图片链路

流程：

1. 选中文件
2. 生成 `Object URL`
3. 预览展示
4. 发送时转成 `Data URL`
5. 组装为 `image_url` message part

为什么不直接预览 `Data URL`：

- `Object URL` 更适合浏览器本地预览，性能更好
- `Data URL` 更适合接口传输

### 10.3 文本文件链路

流程：

1. 校验类型
2. 校验大小
3. 校验数量
4. 进入 `parsing`
5. 异步读取文本
6. 成功更新为 `ready`
7. 失败更新为 `error`

### 10.4 为什么要做上下文预算截断

模型上下文长度有限。

如果用户一次上传多个长文件，直接全部拼进去会导致：

- 请求体过长
- Token 开销过高
- 甚至被服务端拒绝

所以项目引入 `MAX_FILE_CONTEXT_CHARS` 控制预算，并在截断时给出提示。

### 10.5 为什么要做结构化 Prompt 注入

模型更容易理解这种结构：

```text
[文件 1]
文件名：...
类型：...
内容：
...
```

而不是简单把多个文件内容裸拼在一起。

这体现的是前端也在参与“模型输入设计”。

## 11. Markdown 渲染链路

### 11.1 为什么不用简单的纯文本展示

AI 输出通常包含：

- 标题
- 列表
- 代码块
- 表格

如果直接当纯文本渲染，可读性会差很多。

### 11.2 当前项目为什么选择 `marked + DOMPurify + highlight.js`

原因有四个：

1. 当前项目更适合字符串 HTML 渲染管线
2. 流式聊天场景下，两阶段渲染更自然
3. 渲染后 DOM 增强（复制按钮）更方便
4. 动态导入边界清晰

### 11.3 渲染流程

1. `marked` 把 Markdown 转成 HTML
2. `DOMPurify` 净化 HTML
3. React 把 HTML 插入页面
4. `highlight.js` 对代码块高亮
5. 给代码块注入复制按钮

### 11.4 为什么流式期间不做完整代码增强

因为频繁重写代码块 DOM 会导致：

- 打字机抖动
- 滚动位置不稳定
- 高亮反复执行

所以项目区分：

- 流式阶段：优先保证文本平滑更新
- 非流式阶段：再做高亮和复制按钮增强

这是一个很典型的“分阶段渲染优化”。

## 12. 性能、稳定性与资源管理

### 12.1 打字机为什么不是简单逐字输出

项目实现了一个“追赶式打字机”：

- `targetText`：服务端已经返回的完整文本
- `shownText`：当前 UI 已经显示的文本

如果服务端一次吐出大段内容：

- UI 不会傻等一字一字慢慢补
- 会根据 backlog 自动加快展示速度

这样既有打字感，又不会拖太久。

### 12.2 为什么要显式回收 `Object URL`

上传图片预览使用了：

```ts
URL.createObjectURL(file);
```

如果不调用：

```ts
URL.revokeObjectURL(url);
```

浏览器会一直保留这块内存引用。

项目在这些场景里做了回收：

- 删除单张图片
- 清空上传图片
- 删除会话
- 清空会话
- 组件卸载

### 12.3 为什么每个会话单独维护 `AbortController`

因为多会话下，停止生成应该是“按会话停止”，而不是全局停止。

所以项目用：

```ts
abortControllers: Record<string, AbortController | null>;
```

按会话记录控制器。

### 12.4 为什么首页跳聊天页要做幂等控制

如果不做幂等控制：

- 首页透传的草稿可能被多次消费
- 刷新页面可能重复发送同一条消息

所以通过 `useRef token` 机制保证：

- 某次路由状态只消费一次

## 13. 高频面试问题与回答思路

### 13.1 这个项目为什么叫“多会话聊天工作台”

回答思路：

- 不是单个聊天窗口，而是多个独立会话的集合
- 每个会话有自己的消息、历史、草稿、上传资源和流式状态
- 侧栏、路由、持久化都围绕会话进行组织
- 所以更像“工作台”，不是单页聊天 Demo

### 13.2 多会话体现在哪里

可以从三层回答：

1. 数据层：`conversations + orderedConversationIds`
2. 路由层：`/chat/:conversationId`
3. 交互层：侧栏切换、删除、重命名、独立草稿、独立上传

### 13.3 为什么 UI 消息和模型上下文分开存

回答：

- UI 层需要打字机、占位消息、错误提示
- 模型上下文需要干净、稳定
- 如果混在一起，前端展示状态会污染模型历史

### 13.4 为什么首页不直接请求模型

回答：

- 首页没有稳定会话上下文
- 真正请求发生在聊天页更合理
- 这样会话、路由、持久化和上传状态才能统一落在同一个上下文中

### 13.5 为什么选 Zustand 而不是 Redux

回答：

- 项目规模适中
- 更看重轻量与开发效率
- 状态和动作写在一起更适合这个项目
- 不需要 Redux 那套更重的样板代码

### 13.6 为什么用 `marked + DOMPurify + highlight.js` 而不是 `react-markdown`

回答：

- 当前项目是流式聊天场景，不是富文档站点
- 更适合“Markdown -> HTML -> DOM 增强”的两阶段管线
- 渲染后代码块增强和复制按钮注入更自然
- 动态导入边界也更清晰

### 13.7 这个项目最难的点是什么

可以答：

- 不是某个单点技术，而是多会话、流式渲染、文件上传、Markdown 渲染和模型接入叠加后，如何保证链路一致性和状态隔离

### 13.8 如果继续优化，你会做什么

可选回答：

- 增加会话搜索和置顶
- 增加会话归档
- 增加更细粒度的消息级操作
- 对 Markdown 渲染做缓存
- 为服务层增加端点健康度统计
- 增加更完整的错误埋点和性能埋点
