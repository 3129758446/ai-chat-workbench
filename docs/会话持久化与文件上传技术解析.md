# 会话持久化与文件上传技术解析

## 文档目标

这份文档面向两个场景：

1. 学习项目时，快速理解“会话持久化”和“文件上传”到底做了什么。
2. 面试复盘时，能够从业务目标、技术方案、实现细节和设计取舍四个层面说明清楚。

本文会结合当前项目实现，拆解以下两个能力：

1. 多会话聊天的本地持久化。
2. 图片与文本文件上传，并把内容接入模型问答链路。

---

## 一、先理解这两个功能的业务作用

### 1. 会话持久化的作用

会话持久化的核心价值，不是“把数据存起来”这么简单，而是为了让聊天产品具备接近真实 AI 助手的连续体验。

它解决了几个实际问题：

1. 用户刷新页面后，历史会话不丢失。
2. 不同会话之间上下文隔离，不会互相串话。
3. 每个会话都有独立的输入草稿、消息列表、模型上下文和上传草稿。
4. 路由地址和会话 ID 绑定后，某个会话可以被稳定访问和恢复。
5. 侧栏可以展示会话标题、最近摘要、最近更新时间，形成完整的聊天工作台。

换句话说，会话持久化让“单轮对话页面”升级成了“可管理的多会话 AI 应用”。

### 2. 文件上传功能的作用

文件上传能力的核心价值，是把“用户脑子里的问题”升级成“用户的资料 + 用户的问题”。

它解决了几个实际问题：

1. 用户不需要手动复制长文本到输入框。
2. 模型可以基于上传的文本文件回答问题、总结内容、解释代码或分析数据。
3. 图片可以继续走多模态识别能力，文本文件可以走上下文拼接能力。
4. 上传内容跟会话绑定，切换会话时不会互相污染。

本项目当前不是完整 RAG，也不是服务端文件管理系统，而是一个“轻量前端文件解析 + 会话绑定 + 模型上下文拼接”的方案。

---

## 二、整体架构怎么拆

从职责上看，这两个功能分别落在四层：

### 1. 类型层

定义消息、会话、上传文件、上传图片等领域模型。

例如：

1. `Conversation`
2. `ApiMessage`
3. `UiMessage`
4. `UploadingImage`
5. `UploadingTextFile`

### 2. 状态层

使用 Zustand 管理全局聊天状态，并通过 `persist` 做本地持久化。

这一层负责：

1. 创建、切换、删除、重命名会话。
2. 维护每个会话的消息、历史、输入框草稿。
3. 维护每个会话的上传图片和上传文件。
4. 维护流式请求状态和中断控制器。

### 3. 页面编排层

`App.tsx` 负责把路由、store、上传、发送、清理等能力串起来。

这一层负责：

1. 识别当前会话。
2. 处理首页和聊天页两种模式。
3. 处理文件选择、文件分类、上传后跳转。
4. 把状态和事件透传给 `Composer`、`ChatPanel`、`ConversationSidebar`。

### 4. 工具和服务层

工具层负责处理上传文件和消息格式转换，服务层负责请求模型接口。

主要包括：

1. `fileUpload.ts`：校验、解析、上下文拼装。
2. `messageContent.ts`：把文本、文件、图片组装成模型可用的消息内容。
3. `api.ts`：发起流式请求，按是否含图自动切换模型。

## 三、会话持久化功能详解

## 1. 核心目标

会话持久化的目标可以概括为一句话：

让会话在“跨组件、跨路由、跨刷新”之后仍然保持一致。

也就是说，它不仅要解决组件间共享状态问题，还要解决浏览器刷新后的状态恢复问题。

## 2. 核心数据结构

当前项目中，会话模型大致如下：

```ts
interface Conversation {
  id: string;
  title: string;
  createdAt: number;
  updatedAt: number;
  lastMessagePreview: string;
  draftInput: string;
  messages: UiMessage[];
  chatHistory: ApiMessage[];
  uploadingImages: UploadingImage[];
  uploadingFiles: UploadingTextFile[];
  isStreaming: boolean;
}
```

这里最关键的是要理解两套消息：

### 1. `messages`

这是给页面展示用的 UI 消息。

特点是：

1. 适合直接渲染在聊天面板。
2. 可以支持“助手占位消息 + 流式逐步更新文本”。
3. 可以保存预览用的文本和图片片段。

### 2. `chatHistory`

这是给模型接口发送的历史消息。

特点是：

1. 更接近接口协议。
2. 可以是纯字符串，也可以是 `text + image_url` 的多段结构。
3. 决定模型的上下文连续性。

这是一种非常好的设计：UI 展示数据和 API 请求数据分轨管理，避免两边互相牵制。

## 3. 为什么用 Zustand + persist

这套项目选择 Zustand 的原因主要有三点：

1. API 简洁，适合中小型 React 项目。
2. 状态和操作方法可以写在一起，方便维护。
3. `persist` 中间件可以低成本接入本地持久化。

核心结构大致是：

```ts
create<ChatState>()(
  persist(
    (set, get) => ({
      ...
    }),
    {
      name: CHAT_STORE_STORAGE,
      storage: createJSONStorage(() => localStorage),
      partialize: ...,
      merge: ...,
    },
  ),
)
```

这里有三个很重要的面试点：

### 1. `storage`

使用 `localStorage` 作为持久化介质，所以数据会在刷新后保留。

### 2. `partialize`

它决定哪些状态被持久化，哪些状态不被持久化。

### 3. `merge`

它决定持久化数据恢复后，如何和当前运行时默认状态合并。

这也是很多候选人容易忽略的点：持久化不是简单“读出来就用”，而是需要做恢复阶段的规范化。

## 4. 持久化了什么，为什么

当前项目持久化了这些内容：

1. `theme`
2. `currentConversationId`
3. `orderedConversationIds`
4. `conversations`
   - `id`
   - `title`
   - `createdAt`
   - `updatedAt`
   - `lastMessagePreview`
   - `draftInput`
   - `messages`
   - `chatHistory`

这样做的结果是：

1. 刷新后还能恢复当前会话。
2. 侧栏顺序不会乱。
3. 输入框未发送的文字草稿也能恢复。
4. 历史问答可以继续作为上下文使用。

## 5. 为什么有些状态不能持久化

以下状态没有进入持久化：

1. `abortControllers`
2. `uploadingImages`
3. `uploadingFiles`
4. `isStreaming`

原因分别是：

### 1. `AbortController`

它是运行时对象，不能直接序列化，也没有恢复意义。

### 2. `uploadingImages`

图片里包含原始 `File` 和 `object URL`，这些对象刷新后失效，不适合持久化。

### 3. `uploadingFiles`

文本文件里包含 `File` 原始对象，也不适合直接序列化保存；同时这类草稿通常属于“待发送状态”，刷新后恢复反而容易引发认知混乱。

### 4. `isStreaming`

流式请求是临时运行态，刷新页面后请求上下文已经断开，强行恢复会造成状态不一致。

这体现了一个很重要的设计思想：

持久化的是“业务状态”，而不是“运行时对象”。

## 6. 会话是怎么恢复的

项目在恢复持久化数据时，不是直接把本地 JSON 塞回 store，而是做了标准化处理：

1. 遍历持久化的 `conversations`。
2. 通过 `normalizePersistedConversation()` 补齐默认字段。
3. 重置 `uploadingImages`、`uploadingFiles`、`isStreaming`。
4. 为每个会话重新生成 `abortControllers[id] = null`。

这一步非常关键，因为持久化版本和当前运行时代码可能并不完全一致。

如果不做 normalize，后续新增字段、结构变更、版本升级时就容易出 bug。

## 7. 会话生命周期

### 1. 创建会话

`createConversation()` 的逻辑是：

1. 生成唯一 ID。
2. 创建默认会话对象。
3. 插入 `orderedConversationIds` 头部。
4. 设为当前会话。
5. 初始化对应的 `abortControllers`。

### 2. 切换会话

`switchConversation(id)` 只切换当前 ID，页面所展示的消息、草稿、上传预览都会跟着当前会话对象变化。

### 3. 删除会话

删除时做了几件事：

1. 释放图片对象 URL。
2. 中止该会话可能仍在进行中的请求。
3. 删除会话数据。
4. 删除对应的中断控制器。
5. 调整当前会话指针。

### 4. 清空会话

清空不是删除整个对象，而是保留会话身份和标题，重置消息、上下文和上传草稿。

这种设计的好处是：

1. UI 操作更平滑。
2. 不会意外破坏路由和侧栏结构。
3. 可以理解为“保留会话壳，清空会话内容”。

## 8. 路由和持久化的关系

当前聊天页路由是：

```text
/chat/:conversationId
```

这样设计的好处非常明显：

1. URL 直接表达“当前正在看的会话是谁”。
2. 刷新后可以根据路由参数恢复到指定会话。
3. 可以支持从首页跳转到新建会话。
4. 会话切换不再依赖单纯的临时组件状态。

这一点在面试里可以总结为：

通过“路由参数 + 全局 store”共同约束当前会话，让页面地址和业务状态保持一致。

## 9. 标题和摘要是怎么来的

项目没有要求用户必须手动起标题，而是做了自动生成：

1. 从首条用户消息提取文本。
2. 截断成较短标题。
3. 从最后一条历史消息推导摘要。
4. 更新 `updatedAt`。

这套逻辑封装在 `deriveConversationPatch()` 中。

这说明作者不仅考虑了“能不能聊”，还考虑了“侧栏可读性”和“会话管理体验”。

## 四、文件上传功能详解

## 1. 当前支持的上传能力

这个项目的上传能力并不是只有图片。

它实际上支持两类资源：

1. 图片文件
2. 文本类文件

文本类文件包括：

1. `txt`
2. `md`
3. `markdown`
4. `json`
5. `csv`
6. `log`
7. `js`
8. `ts`
9. `tsx`
10. `jsx`
11. `css`
12. `html`
13. `xml`
14. `yaml`
15. `yml`

这说明它不是传统“上传到服务器”的文件上传，而是“浏览器读取文件内容后，参与模型问答”的上传。

## 2. 上传入口在哪里

上传入口在 `Composer.tsx`：

```html
<input
  type="file"
  accept="image/*,.txt,.md,.markdown,.json,.csv,.log,..."
  multiple
  hidden
  onChange={onFileChange}
/>
```

这里有几个关键点：

1. 使用隐藏的 `input[type=file]`，由按钮手动触发。
2. `multiple` 表示支持多文件选择。
3. `accept` 同时包含图片和文本文件。
4. 真正的业务逻辑不放在组件里，而是通过 `onFileChange` 交给上层处理。

这是典型的“展示组件只负责交互，业务编排放在容器层”的 React 设计。

## 3. 文件选择之后发生了什么

上传主流程在 `App.tsx` 的 `handleFileChange()` 中。

大致过程如下：

### 第一步：拿到用户选择的文件列表

```ts
const files = Array.from(event.target.files || []);
```

### 第二步：按类型拆分成图片和文本文件

```ts
const nextImages = files.filter(isImageFile)
const textFileInputs = files.filter((file) => !isImageFile(file))
```

### 第三步：确定这些上传草稿属于哪个会话

规则是：

1. 如果当前已经在聊天页，就绑定到当前会话。
2. 如果当前在首页，就先创建一个新会话，再把上传草稿挂到这个新会话上。

这个细节很重要，因为它解决了首页上传后会话归属不明确的问题。

### 第四步：对文本文件做数量和类型校验

包括：

1. 单会话最多 `3` 个文本文件。
2. 单文件大小不超过 `1MB`。
3. 必须是受支持的文本类型。

校验失败不会直接丢弃文件，而是生成一个 `status = "error"` 的文件卡片，并带上错误信息。

这是一个很好的交互设计，因为用户能立刻知道为什么失败。

### 第五步：把图片和文本草稿写入当前会话

图片进入 `uploadingImages`，文本文件进入 `uploadingFiles`。

### 第六步：异步解析文本文件内容

对于 `parsing` 状态的文本文件，会调用 `parseTextFile(file.file)`：

1. 成功后更新为 `ready` 并写入 `text`。
2. 失败后更新为 `error` 并写入错误信息。

### 第七步：首页模式下跳转到聊天页

如果上传发生在首页，会跳转到：

```text
/chat/:targetConversationId
```

这样用户就能继续在新会话里基于这些文件提问。

## 4. 图片和文本文件为什么分开处理

因为两者最终进入模型的方式完全不同。

### 1. 图片

图片不会先转成可阅读文本，而是：

1. 在前端生成预览 URL 用于 UI 展示。
2. 在发送时读取原始 `File`。
3. 转成 `data URL`。
4. 作为 `image_url` 片段传给多模态模型。

### 2. 文本文件

文本文件不会转成图片，也不会上传到服务器，而是：

1. 在浏览器里读取文本。
2. 拼成结构化说明文本。
3. 作为用户消息的文本上下文传给模型。

这体现的是“同一个上传入口，两个不同处理通道”的设计。

## 5. 文本文件校验与解析

`fileUpload.ts` 里集中做了这几件事：

### 1. 类型识别

通过“扩展名 + MIME”双重判断，而不是只看某一个字段。

原因是：

1. 浏览器返回的 MIME 可能为空。
2. 不同系统或浏览器对同一文件类型的 MIME 可能不一致。
3. 只信任扩展名也不稳妥。

这是一种典型的健壮性设计。

### 2. 大小限制

```ts
MAX_TEXT_FILE_SIZE = 1024 * 1024
```

也就是单文件最大 `1MB`。

限制大小的原因包括：

1. 避免浏览器读大文件造成卡顿。
2. 避免模型上下文被单个文件吃满。
3. 降低发送和序列化成本。

### 3. 数量限制

```ts
MAX_TEXT_FILE_COUNT = 3
```

这是为了控制会话复杂度和上下文长度。

### 4. 文件读取

```ts
const text = await file.text();
```

优点是实现简单，不需要额外依赖。

同时还做了空文件判断：

1. 浏览器不支持 `file.text()` 时直接报错。
2. 文件内容为空时直接报错。

## 6. 文件状态模型为什么重要

文本文件状态分为三种：

```ts
type UploadingFileStatus = "parsing" | "ready" | "error";
```

意义如下：

1. `parsing`：正在异步读取中。
2. `ready`：解析成功，可以参与发送。
3. `error`：解析失败，只展示，不参与发送。

这个状态模型的价值是：

1. UI 可以准确展示文件当前状态。
2. 发送按钮可以基于业务规则判断是否可发送。
3. 解析失败不会污染模型上下文。

这是典型的前端异步任务状态建模能力，面试里很加分。

## 7. 文件如何进入模型上下文

这是整个上传能力最核心的一步。

发送时，`useSendMessage()` 会拿到当前会话中所有 `ready` 的文本文件：

```ts
const readyFiles = uploadingFiles.filter((file) => file.status === "ready");
```

然后调用：

```ts
buildUserMessageContent(text, images, files)
```

其中对文本文件的处理是：

1. 先调用 `buildFileQuestionText(question, files)`。
2. 将文件内容拼成一个结构化文本块。
3. 再把“用户问题”拼到最后。

最终结构类似：

```text
以下是用户上传的文件内容，请基于这些内容回答问题。

[文件 1]
文件名：demo.ts
类型：application/typescript
内容：
...

用户问题：请解释这段代码的作用
```

这一步本质上是在做：

把“文件”转换成“模型可以理解的自然语言上下文”。

## 8. 为什么要做内容截断

项目里设置了：

```ts
MAX_FILE_CONTEXT_CHARS = 20_000
```

也就是说，多个文件拼接后，进入模型的总文本上下文有长度上限。

这样做的原因有三点：

1. 避免 prompt 过长影响响应速度和成本。
2. 避免模型上下文被文件内容挤占，导致用户问题关注度下降。
3. 保持前端方案轻量，不演变成复杂的检索系统。

截断时还会明确补一个提示：

```text
注意：由于上下文长度限制，部分文件内容已截断。
```

这个提示非常重要，因为它能让用户理解回答为什么可能不完整。

## 9. 发送时如何同时兼容文本、文件、图片

这是这套实现最值得学习的地方之一。

`buildUserMessageContent()` 的策略是：

### 情况 1：没有图片

直接返回文本字符串，这个文本里已经包含了文件内容和用户问题。

### 情况 2：有图片

返回 `MessagePart[]`，格式如下：

1. 先放 `text` 片段。
2. 再按顺序放多个 `image_url` 片段。

也就是：

```ts
[
  { type: "text", text: "...文件上下文 + 用户问题..." },
  { type: "image_url", image_url: { url: "data:..." } },
  { type: "image_url", image_url: { url: "data:..." } },
]
```

这样做的优点是：

1. 不破坏原有多模态协议。
2. 可以让模型先理解文字任务，再看图片。
3. 文件上下文和图片输入可以共存。

## 10. 为什么图片预览要用 `URL.createObjectURL`

图片上传后会立刻生成本地预览地址：

```ts
URL.createObjectURL(file)
```

这样可以做到：

1. 不上传服务器也能立刻预览。
2. 不需要先把整张图片转成 base64 才能显示。
3. UI 反馈快，内存和性能更友好。

但对应也要做清理：

```ts
URL.revokeObjectURL(url)
```

项目在以下时机会释放 URL：

1. 移除单张图片时。
2. 清空当前会话时。
3. 删除会话时。
4. 组件卸载时做兜底清理。

这反映了良好的资源管理意识，面试里可以重点讲。

## 五、发送链路中的协同关系

这两个功能不是孤立存在的，它们最终都汇入消息发送链路。

完整流程可以概括为：

1. 用户在当前会话输入文本或上传文件。
2. 上传资源先挂在当前会话的草稿区。
3. 点击发送后，读取当前会话的文本、图片、已就绪文件。
4. 构造 `userContent`。
5. 先更新 UI 消息。
6. 再写入 `chatHistory`。
7. 再清空当前会话的输入草稿和上传草稿。
8. 插入 assistant 占位消息。
9. 发起流式请求。
10. 持续更新 assistant 文本。
11. 请求成功后把 AI 回复写回 `chatHistory`。

这里体现了三个很重要的工程思想：

### 1. 先更新 UI，再请求接口

用户点击发送后马上看到自己的消息，体验更自然。

### 2. 发送使用快照

发送前复制一份 `images` 和 `files`，即使随后清空草稿，也不会影响本次请求。

### 3. 成功与失败都要回收状态

无论请求是否成功，都要在 `finally` 里恢复：

1. `AbortController`
2. `isStreaming`

这是一种很规范的异步流程控制方式。

## 八、这套方案的局限性

这也是面试里很值得主动提到的部分。

## 1. 文件能力本质上还是 Prompt 拼接

它不是检索增强，不具备：

1. 语义召回
2. 分片检索
3. 向量索引
4. 大文件精准定位

所以文件一长，就只能截断。

## 2. 未发送文件草稿不持久化

这虽然是合理选择，但用户刷新后上传草稿会消失。

## 3. 文本文件解析能力有限

目前只适合纯文本类文件，不支持：

1. PDF
2. Word
3. Excel
4. OCR

## 4. 会话持久化还停留在本地

如果用户换设备、清缓存或换浏览器，数据无法同步。

## 5. 历史上下文可能越来越长

随着 `chatHistory` 累积，请求成本和上下文污染风险会增加，后续可能需要做摘要或裁剪策略。

## 十、面试怎么讲这两个功能

下面给一个比较适合面试表达的版本。

### 1. 一分钟版本

我在项目里做了两个比较完整的能力：多会话持久化和文件上传问答。

多会话持久化这块，我用 Zustand 做全局聊天状态管理，用 `persist` 把会话列表、当前会话、消息历史和输入草稿存到 `localStorage`。为了避免恢复异常，我没有直接反序列化后使用，而是在 `merge` 阶段对会话做标准化，重置了 `AbortController`、上传草稿和流式状态这些运行时对象。

文件上传这块，我把图片和文本文件放在同一个上传入口里处理。图片走多模态链路，发送时转成 `image_url`；文本文件在前端用 `File.text()` 读取，经过大小、数量、类型校验后，拼成结构化上下文加入用户问题。这样用户可以直接基于上传的文件继续提问，不需要手动复制内容。

### 2. 三分钟版本的关键词

可以重点展开以下关键词：

1. 路由和会话 ID 绑定。
2. UI 消息与 API 历史分离。
3. 持久化白名单和恢复标准化。
4. 上传资源按会话隔离。
5. 文本文件是前端解析，不是服务端上传。
6. 图片和文本文件走不同的消息构造通道。
7. 流式消息用占位消息 + 增量更新实现。
8. 对象 URL 及时释放，避免内存泄漏。

### 3. 可能被追问的问题

#### 问：为什么不把所有状态都持久化？

答：因为 `AbortController`、`File`、对象 URL、流式状态这些都是运行时对象，序列化没有意义，而且刷新后也无法恢复真实执行现场，所以只持久化业务状态，不持久化运行时状态。

#### 问：为什么文本文件不直接上传到后端？

答：当前需求更偏轻量文件问答，不需要文件管理、对象存储和检索能力。前端直接读取文本后拼入上下文，开发成本低，能快速验证业务价值。

#### 问：为什么要区分 `messages` 和 `chatHistory`？

答：因为 UI 展示和接口协议的关注点不同。UI 需要支持占位消息、预览和渐进更新；接口历史需要严格符合模型输入结构。分离后可维护性更好。

#### 问：如果文件很大怎么办？

答：当前方案通过文件大小、数量和上下文总长度做限制，并在超长时截断提示。如果业务继续发展，就要升级到分片、检索和摘要方案。

#### 问：图片为什么不用对象 URL 直接发给模型？

答：对象 URL 只是浏览器本地预览地址，模型接口无法直接访问，所以发送时需要把图片文件转成 data URL，再按多模态协议传给接口。

## 十二、最后总结

1. 用领域模型统一描述消息、会话、上传文件。
