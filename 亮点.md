这个项目是一个“前端直连大模型接口（开发态通过 Vite 代理同源转发）+ 流式聊天 UI”的完整闭环。下面按真实代码链路，给你做一版从启动到响应渲染的全流程拆解。

一、整体分层

入口与路由层：负责页面挂载和 home/chat 两种模式切换，见 main.tsx、index.tsx、router.tsx
页面编排层：统一连接状态、组件、发送逻辑、副作用，见 App.tsx
状态层（Zustand）：保存输入、消息、历史、上传图片、流式状态、中断控制器，见 chatStore.ts
发送业务层：把“发送一次消息”变成标准流程，见 useSendMessage.ts
副作用层：主题、滚动、输入框自适应、URL 资源清理，见 useAppEffects.ts
服务层：端点选择、模型选择、SSE/JSON 解析、失败重试，见 api.ts
展示层组件：欢迎区、聊天面板、输入区，见 WelcomeSection.tsx、ChatPanel.tsx、Composer.tsx
工具与类型层：Key 处理、Markdown 渲染、图片编码、类型约束，见 helpers.ts、markdown.ts、messageContent.ts、chat.ts
二、应用启动流程

浏览器加载页面后，React 根节点在 main.tsx 被挂载。
路由提供器在 index.tsx 启动。
路由规则在 router.tsx 生效：
/ 对应首页模式（App mode=home）
/chat 对应聊天模式（App mode=chat）
其他路径重定向到 /
App 初始化时从 Zustand 读取全局状态，挂载副作用，绑定发送函数，见 App.tsx。
三、核心状态模型（最关键）
在 chatStore.ts 里有两套“消息概念”：

messages：UI 展示消息（带 id，适合渲染和流式覆盖）
chatHistory：发给模型的上下文消息（严格按模型输入结构）
这两个分离是项目质量很高的一点：

UI 关注“怎么显示”
历史关注“怎么喂给模型”
两者分离后，流式显示、错误替换、中断回滚都更容易处理
同时 store 还管理：

input：输入框文字
uploadingImages：待发送图片（含预览 URL 和 File）
isStreaming：当前是否在生成
abortController：用于停止请求
theme：明暗主题并持久化到 localStorage
四、从用户点击发送到看到回复：完整主链路
这条链路集中在 useSendMessage.ts：

模式判断
如果当前在 home，只做跳转到 /chat，并把草稿和自动发送标记放进路由 state，不直接请求接口。
并发保护
如果 isStreaming 为 true，直接拒绝再次发送，避免重入。
内容校验
文本和图片都为空则不发送。
API Key 检查
先从 localStorage 取，见 helpers.ts。
没有就弹窗让用户输入，输入后写回本地。
组装用户消息内容
无图时直接用纯文本。
有图时把每张图片转 data URL，组装为多模态片段数组，见 messageContent.ts。
先写 UI 再请求
先 addUiMessage 写入用户消息，用户立即看到自己发出的内容。
同时 pushHistory，把用户消息加入模型上下文。
清空输入和上传队列
setInput 置空，clearUploadingImages 清理待发图片并释放 URL。
插入一条空 assistant 消息占位
后续每次收到增量就覆盖这条消息，实现打字机效果。
创建 AbortController 并进入流式状态
setAbortController 和 setStreaming(true)。
发起流式请求
调用 streamChatCompletion，传入 onDelta 回调。
onDelta 每来一次，就 updateUiMessageText 覆盖 assistant 占位文本。
完成后入历史
把最终 assistant 文本 pushHistory，形成下一轮上下文。
错误分级处理
AbortError：用户主动停止，回滚本轮 user 历史，避免污染上下文。
401/403：清理本地 Key，提示重新配置。
429：提示限流或额度问题。
400 且有图片：提示视觉模型能力或配置问题。
Failed to fetch：提示网络/代理问题。
finally 中统一恢复 setStreaming(false) 和清空 controller。
五、接口与流式解析细节
在 api.ts：

模型自动切换
消息里有 image_url 就用视觉模型，否则用文本模型。
端点策略
默认端点来自 constants.ts，优先国内再国际回退。
你现在改成了同源路径（/api、/api-intl），由代理转发。
自定义端点会做同源校验，外域会被拒绝并回退默认端点。
请求解析分支
HTTP 非 2xx：抛 ApiError（附 status 和 endpoint）
body 为空：抛错
content-type 是 JSON：一次性取完整内容
否则按 SSE 读取流：逐行找 data:，忽略非数据行，解析 delta 累加并回调 onDelta
多端点重试
只对传输层错误做回退（如 Failed to fetch、NetworkError）
业务错误（如 401、429）不盲目重试，直接抛出
六、UI 展示是怎么跟流式绑定的

聊天面板在 ChatPanel.tsx
assistant 消息用 Markdown 渲染，user 消息按文本+图片直出
只有“最后一条 assistant 且当前 isStreaming”会显示打字状态，避免误伤历史消息
Markdown 渲染链路在 markdown.ts：
marked 解析
DOMPurify 净化（防 XSS）
enhanceCodeBlocks 做代码高亮和复制按钮增强
七、页面副作用与体验细节
在 useAppEffects.ts：

主题同步：写 body 的 data-theme，切换高亮 CSS 文件启停
输入框自适应：按 scrollHeight 自动长高，上限 160px
自动滚动：消息或上传变化后滚到底部
资源清理：卸载时 revokeObjectURL，避免内存泄漏
八、样式组织方式

单入口样式在 index.css，按顺序导入：
主题变量
基础样式
欢迎区
聊天区
输入区
响应式
这种顺序保证变量和基础规则先加载，后面模块样式更稳定。
